1. Dubbo概述

1.1 Dubbo概念

Dubbo是阿里巴巴公司开源的一个高性能、轻量级的 Java RPC 框架。

致力于提供高性能和透明化的 RPC 远程服务调用方案，以及 SOA 服务治理方案。

官网：http://dubbo.apache.org

1.2 Dubbo架构

2. Dubbo快速入门

使用dubbo前首先要选取注册中心Registry，这里选择zookeeper作为注册中心，下载安装启动zookeeper

步骤

①创建服务提供者Provider模块(dubbo-service)

②创建服务消费者Consumer模块(dubbo-web)

③在服务提供者模块编写 UserServiceImpl 提供服务

④在服务消费者中的 UserController 远程调用UserServiceImpl 提供的服务

⑤分别启动两个服务，测试

2.1 maven分模块

①创建服务提供者Provider模块(dubbo-service) jar

②创建服务消费者Consumer模块(dubbo-web) war

③在服务提供者模块编写 UserServiceImpl 提供服务

④在服务消费者Consumer模块依赖服务提供者Provider模块

⑤分别启动两个服务，测试

2.2 分布式

分布式会将不同的模块运行在不用的机器上，需要各模块都能对外服务所以两个模块均要打war包，通过Dubbo进行RPC远程通信

dubbo配置

<!--修改端口 默认端口20880-->
<!--<dubbo:protocol port="20880"/>-->

<!--dubbo的配置-->
<!--1.配置项目的名称-->
<dubbo:application name="dubbo-service"/>
<!--2.配置注册中心的地址-->
<dubbo:registry address="zookeeper://192.168.5.130:2181"/>
<!--3.配置dubbo包扫描-->
<dubbo:annotation package="com.example.service.impl"/>


<!-- 元数据配置 -->
<dubbo:metadata-report address="zookeeper://192.168.5.130:2181" />

步骤

①创建服务提供者Provider模块(dubbo-service) war

②创建服务消费者Consumer模块(dubbo-web) war

③在服务提供者模块编写 UserServiceImpl 提供服务

④在服务消费者中的 UserController 远程调用UserServiceImpl 提供的服务

⑤分别启动两个服务，测试

此时会发现一个问题，两个模块部署在两台机器上无法依赖，取消依赖服务消费者中的接口又会找不到而报错，

所以解决办法是将公共的接口抽出来，让服务提供者Provider模块和服务消费者Consumer模块都依赖公共接口模块(dubbo-interface)

注解说明:

@Service

名称 @Service
作用 将这个类提供的方法（服务）对外发布。将访问的地址 ip 端口 路径注册到注册中心中
使用对象 消息提供方的类/接口

@Reference

名称 @Reference
作用 1.从zookeeper注册中心获取userService的访问url
2.进行远程调用RPC
3.将结果封装为一个代理对象。给变量赋值
使用对象 消息消费方所要调用的接口变量

3. Dubbo-admin安装

3.1 环境准备

node.js

https://nodejs.org/en/

maven

3.2 下载Dubbo-Admin

https://github.com/apache/dubbo-admin

3.3 解压zip包并修改配置文件

解压后我们进入…\dubbo-admin-develop\dubbo-admin-server\src\main\resources目录，找到 application.properties 配置文件 进行配置修改

修改zookeeper地址为自己的zookeeper地址

# centers in dubbo2.7
admin.registry.address=zookeeper://192.168.5.130:2181
admin.config-center=zookeeper://192.168.5.130:2181
admin.metadata-report.address=zookeeper://192.168.5.130:2181

admin.registry.address注册中心
admin.config-center 配置中心
admin.metadata-report.address元数据中心

3.4 打包项目

在 dubbo-admin-develop 目录执行打包命令

mvn clean package -Dmaven.test.skip=true

3.5 启动后端

cd dubbo-admin-distribution/target
java -jar dubbo-admin-${project.version}.jar

其中${project.version}为对应的版本号

3.6 启动前端

dubbo-admin-ui 目录下执行命令

npm run dev

3.7 访问给出的地址即可

4. Dubbo高级特性

4.1 序列化

dubbo 内部已经将序列化和反序列化的过程内部封装了

我们只需要在定义pojo类时实现Serializable接口即可

一般会定义一个公共的pojo模块，让生产者和消费者都依赖该模块

如上例可以新建一个dubbo-pojo模块

4.2 地址缓存

注册中心挂了，服务仍可以正常访问。

因为dubbo服务消费者在第一次调用时，会将服务提供方地址缓存到本地，以后在调用则不会访问注册中心。

当服务提供者地址发生变化时，注册中心会通知服务消费者

4.3 超时与重试

超时

  • 服务消费者在调用服务提供者的时候发生了阻塞、等待的情形，这个时候，服务消费者会一直等待下去。

  • 在某个峰值时刻，大量的请求都在同时请求服务消费者，会造成线程的大量堆积，势必会造成雪崩。

  • dubbo 利用超时机制来解决这个问题，设置一个超时时间，在这个时间段内，无法完成服务访问，则自动断开连接。

  • 使用timeout属性配置超时时间，默认值1000，单位毫秒。

其中timeout属性可以配置在@Service或者@Reference均可，但是建议配置在@Service中，因为服务的提供方可以预估出来业务的执行时间。

@Service(timeout = 3000,retries = 2)//当前服务3秒超时，重试2次，一共3次

@Reference(timeout = 1000,retries = 1)//当前服务1秒超时，重试1次，一共2次

@Service或者@Reference同时配置时以@Reference上的为准

重试

  • 设置了超时时间，在这个时间段内，无法完成服务访问，则自动断开连接。

  • 如果出现网络抖动，则这一次请求就会失败。

  • Dubbo 提供重试机制来避免类似问题的发生。

  • 通过 retries 属性来设置重试次数。默认为 2 次。

配置同timeout。

4.4 多版本

灰度发布：当出现新功能时，会让一部分用户先使用新功能，用户反馈没问题时，再将所有用户迁移到新功能。

dubbo 中使用version 属性来设置和调用同一个接口的不同版本

@Service(version = "v1.0")
public class UserServiceImpl implements UserService {}
@Service(version = "v2.0")
public class UserServiceImpl2 implements UserService {}

@Reference(version = "v1.0")//使用v1.0版本
@Reference(version = "v2.0")//使用v2.0版本

4.5 负载均衡

策略 算法 特性
Random RandomLoadBalance 加权随机，按权重设置随机概率。
RoundRobin RoundRobinLoadBalance 加权轮询，按公约后的权重设置轮询比率，循环调用节点
LeastActive LeastActiveLoadBalance 加权最少活跃调用优先，活跃数越低，越优先调用，相同活跃数的进行加权随机。
ShortestResponse ShortestResponseLoadBalance 加权最短响应优先，在最近一个滑动窗口中，响应时间越短，越优先调用。相同响应时间的进行加权随机。
ConsistentHash ConsistentHashLoadBalance 一致性 Hash，相同参数的请求总是发到同一提供者

配置

@Reference(loadbalance = "random")//采用随机策略

4.6 集群容错

模式 说明
Failover Cluster 失败重试。默认值。当出现失败，重试其它服务器 ，默认重试2次，使用 retries 配置。一般用于读操作
Failfast Cluster 快速失败，只发起一次调用，失败立即报错。通常用于写操作。
Failsafe Cluster 失败安全，出现异常时，直接忽略。返回一个空结果。
Failback Cluster 失败自动恢复，后台记录失败请求，定时重发。通常用于消息通知操作。
Forking Cluster 并行调用多个服务器，只要一个成功即返回。
Broadcast Cluster 广播调用所有提供者，逐个调用，任意一台报错则报错。

配置

@Reference(cluster = "failover")//采用失败重试(默认)

4.7 服务降级

mock=force:return null 表示消费方对该服务的方法调用都直接返回null值，不发起远程调用。用来屏蔽不重要服务不可用时对调用方的影响。
mock= fail:return null 表示消费方对该服务的方法调用在失败后，再返回null值，不抛异常。用来容忍不重要服务不稳定时对调用方的影响。

配置

@Reference(mock = "force:return null")
@Reference(mock = "fail:return null")

注意：

  1. 新版本dubbo注解名称不一样 例如@Reference -> @DubboReference
